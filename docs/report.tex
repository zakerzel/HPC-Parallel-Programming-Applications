\begin{abstract}
This report presents four parallel programming applications that cover dense and sparse matrix multiplication, microscopy-style cell image processing, a forest-fire cellular automaton guided by FIRMS-like hotspot data, and K-Means clustering. Each exercise includes a serial baseline and at least one parallel or distributed-memory implementation. Local experiments were executed for the serial and multiprocessing workflows that could be reproduced in the current environment, while the repository also includes MPI code and instructions for running distributed-memory tests on a machine with an MPI runtime.
\end{abstract}
\begin{IEEEkeywords}
High performance computing, multiprocessing, mpi4py, matrix multiplication, image processing, cellular automata, K-Means.
\end{IEEEkeywords}

\section{Introduction}
The repository was organized so that each exercise can be executed independently and compared against its serial baseline. The code is split into \texttt{exercise\_1} through \texttt{exercise\_4}, while the \texttt{docs} directory contains this IEEE-style report and generated figures. The project emphasizes reproducibility by fixing seeds, saving CSV outputs, and preserving generated plots inside each exercise folder.

\section{Exercise 1. Parallel Matrix Multiplication}
\subsection{Evidence of completeness}
The first exercise includes five matrix multiplication strategies in \texttt{exercise\_1/src/matmul\_core.py}: serial dense multiplication, multiprocessing by rows, multiprocessing by columns, multiprocessing by blocks, and a recursive Strassen hybrid. In addition, \texttt{matmul\_mpi.py} provides a distributed-memory version based on scatter/broadcast/gather. Two benchmark scripts were included: one for dense synthetic matrices and one for sparse matrices.

\subsection{Results}
Dense experiments were executed for square matrices of sizes 64, 128, and 256 using four workers. The serial baseline was extremely fast because NumPy delegates the core operation to optimized native libraries, so Python-level multiprocessing mainly added process startup and data-transfer overhead. The row, column, and block decompositions remained correct for all tested sizes, but their runtimes were much slower than the baseline. The Strassen implementation remained correct and became faster than the multiprocessing variants because it stayed inside a single process and switched to the optimized matrix product at the recursion threshold.

For sparse data, the repository includes a CSR-based experiment that mimics the requirement of comparing dense and sparse behavior. The local results showed that a sparse representation can reduce runtime dramatically when sparsity is exploited directly. For instance, at $n=512$ and density $0.01$, the dense serial path took 0.002761 s, the row-parallel dense path took 1.452383 s, and the CSR multiplication finished in 0.000308 s.

\subsection{Discussion}
This exercise highlights an important HPC lesson: parallelism is not automatically beneficial. When the serial baseline already relies on highly optimized low-level kernels, Python multiprocessing can be dominated by inter-process communication and memory copying. The block-based strategy was conceptually closer to two-dimensional decomposition, but for the tested sizes it still paid a large overhead for splitting and reconstructing arrays. The sparse experiment also showed that choosing the correct data structure can have a much bigger impact than adding parallel workers to an inefficient representation. The repository still includes the MPI implementation because larger matrices and true distributed-memory environments are more appropriate settings for testing scalability.

\section{Exercise 2. Parallel Cell Image Processing and Morphological Characterization}
\subsection{Evidence of completeness}
The second exercise contains a dataset inspection tool, a serial/parallel image processing pipeline, and a visualization script. The processing code reads grayscale microscopy images, performs Otsu thresholding with basic morphological cleanup, labels connected components, and computes object-level descriptors: bounding box, area, major axis length, and minor axis length. Overlays are exported to illustrate the detected regions.

For offline validation, a synthetic microscopy-like image set was generated in the expected folder structure \texttt{exercise\_2/data/DIC-C2DH-HeLa/01}. The same code can be used directly with the real DIC-C2DH-HeLa dataset by replacing the synthetic images with the actual sequence.

\subsection{Results}
The inspection script confirmed a consistent image shape of 256$\times$256 pixels for the local validation set. The serial pipeline processed 10 images in 0.347191 s, while the four-worker multiprocessing version processed the same images in 1.482143 s. The summary tables printed by both runs matched image by image, so the work partition by image preserved the final counts and aggregated descriptors.

\subsection{Discussion}
The results again show that small tasks do not always benefit from multiprocessing. Each image in the validation set was relatively small, so process creation and inter-process serialization cost more than the saved computation time. Even so, the parallel design is technically reasonable because microscopy workloads often involve hundreds or thousands of independent images. Once the dataset size grows, image-level task parallelism becomes more attractive. A second limitation is segmentation quality. The current pipeline uses thresholding and morphological filtering rather than Cellpose, so the accuracy of the geometric descriptors depends strongly on contrast, touching cells, and noise. The repository therefore separates the segmentation stage from the measurement stage so a stronger model can be plugged in later. During execution, \texttt{scikit-image} emitted deprecation warnings for several morphology and region-property aliases; these warnings did not stop execution or change the saved outputs, but they should be cleaned up in a future revision.

\section{Exercise 3. Forest Fire Cellular Automaton Driven by FIRMS Data}
\subsection{Evidence of completeness}
The third exercise includes code for generating a FIRMS-like CSV sample, converting hotspot detections into a regular grid, simulating the automaton in serial, and splitting the domain by rows for MPI execution. The state space uses the values required in the assignment: non-burnable, susceptible vegetation, burning, and burned. The MPI version exchanges ghost rows with neighbor processes at each time step.

\subsection{Results}
A local serial experiment was executed on a 128$\times$128 grid for 20 time steps. The simulation completed in 0.005171 s and exported a GIF plus selected snapshots. Because an MPI runtime was not available in the current execution environment, the distributed-memory benchmark could not be run here, but the repository includes the full \texttt{simulate\_mpi.py} implementation and the exact \texttt{mpiexec} command required to reproduce it on the student machine.

\subsection{Discussion}
The automaton is a simplified model of fire propagation, not a true perimeter reconstruction. FIRMS hotspots indicate satellite detections rather than full fire boundaries, so they are best interpreted as ignition or high-intensity reference points. This makes the model useful for HPC experimentation but limited as a scientific fire-behavior simulator. The row-wise MPI decomposition is appropriate because each iteration only requires boundary exchange with immediate neighbors. The main scaling risks are communication frequency, imbalance caused by uneven ignition density, and the mismatch between simplified rules and the complexity of real fuels, wind, and terrain.

\section{Exercise 4. Parallel K-Means Clustering}
\subsection{Evidence of completeness}
The fourth exercise includes a serial NumPy baseline and an MPI implementation of K-Means. The MPI version distributes the dataset by rows, computes local assignments, and uses collective communication to aggregate per-cluster sums and counts before updating centroids globally. The repository also includes a helper benchmark script and optional support for the Covertype dataset.

\subsection{Results}
The local serial benchmarks were executed on synthetic datasets of increasing size. For 3{,}000 samples with 10 features and $k=3$, the runtime was 0.002326 s with a silhouette score of 0.545747. For 10{,}000 samples with 20 features and $k=5$, the runtime increased to 0.139998 s with inertia 48543.873181 and silhouette 0.501917. A separate serial run on a larger 20{,}000-sample dataset with the same dimensionality completed in 0.061829 s after 4 iterations with inertia 20816.825796 and silhouette 0.766318.

\subsection{Discussion}
K-Means is a strong candidate for distributed-memory parallelization because the assignment step can be computed independently on local blocks and only compact cluster statistics must be synchronized. The communication pattern in the MPI implementation is therefore natural: \texttt{Allreduce} combines sums and counts for each cluster after every iteration. The limitation is that synchronization occurs at every update, so communication cost grows with the number of iterations, processes, and clusters. This means the distributed approach becomes more attractive as the dataset grows, while small synthetic examples may still favor the serial baseline.

\section{Conclusion}
The repository now contains complete source code for the four required exercises, an IEEE-style report, generated CSV outputs, figures, and reproducible commands. The local experiments consistently showed that parallelism must be evaluated in context: Python multiprocessing can be slower than optimized serial kernels on small workloads, yet the same design patterns remain valid and necessary for larger problems and distributed-memory environments. The finished submission therefore satisfies the structure of the assignment while also documenting the observed performance limitations rather than hiding them.
